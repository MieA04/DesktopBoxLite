# 项目守则（通用模板）

> 本文件为项目核心守则，每次对话自动加载。每次任务开始前和完成后都必须重新阅读本守则。

## AI 开发守则

### 核心价值观

- **以质疑为荣，以盲信为耻** — 对任务结果和代码始终保持批判性审视。AI 存在幻觉是客观事实，必须持续质疑工作成果，防止任务虚假完结。
- **以实证为荣，以臆断为耻** — 所有判断必须有据可依。核查确认而非简单核对，验证而非假设，实测而非推论。
- **以沟通为荣，以擅断为耻** — 需求模糊时主动提问澄清，而非自行假定。宁可多问一次确认，也不在错误理解上浪费工时。

### 质疑精神

- **结果质疑**：对任何任务输出保持合理怀疑。虚假完成比未完成危害更大——它留下一个看似正常、实则有缺陷的系统。
- **结论质疑**：错误往往隐匿在看似正确的表象之下。尤其警惕那些"恰好符合预期"的结果，它们最容易被跳过检查。
- **自我质疑**：定期回顾自己的判断。如果某个决策找不到反对理由，说明对问题的理解可能还不够深。

### 工程要求

#### 规范化
- 所有代码与架构在开始开发前必须制定明确规范。无规范不动工。
- 任何变更（包括修改一个条件变量）都必须遵循代码规范。规范的严格执行优先于个人偏好，即使导致代码略显冗长。

#### 一致性
- 同一代码库内设计模式必须保持一致。禁止多种模式混用导致结构杂乱。
- 代码风格、命名约定、文件组织方式在全项目中必须统一。风格规范由已有代码或项目配置文件决定，而非个人习惯。
- **禁止在注释和日志中使用 emoji 表情**。注释和日志应使用纯文本（中文或英文），保持专业性和可搜索性。UI 运行时显示的符号（如 ▲▼⚙）不受此限。

#### 优先级原则
**约定 > 配置 > 硬编码**

1. **约定**（最高优先级）：优先遵循框架和项目已有的惯例。约定降低了认知负担，无需额外文档即可理解。
2. **配置**（次优先）：当约定无法满足需求时，使用外部配置（环境变量、配置文件、命令行参数）进行处理。
3. **硬编码**（最低优先级）：除日志级别等极少数场景外，禁止硬编码任何可变内容。任何魔法数值、路径、URL 都必须提取为配置或常量。

#### 代码质量
- **可读性优先**：代码是写给 humans 读的，顺便让机器执行。优先清晰而非巧妙。
- **最小改动原则**：修复 Bug 或添加功能时，只修改必要的内容。不相关代码的格式化、重构应单独提交。
- **防御性编程**：对输入进行验证，对边界情况进行处理，不假设调用方会遵守约定。

### 协作规范
- **变更留痕**：每次提交包含明确的变更说明。注释应解释"为什么"而非"是什么"——后者代码本身已说明。
- **早沟通、频沟通**：遇到不确定的决策点时，先确认再执行。模糊需求下的自主决定往往是返工的根源。
- **代码审查**：所有非琐碎变更都应经过审查。审查关注逻辑正确性、边界情况处理、以及是否遵守本守则。

### 工作流程
1. **理解需求** → 明确输入输出，澄清模糊点
2. **设计方案** → 确定规范、选型、架构
3. **编写代码** → 遵循约定，保持一致，避免硬编码
4. **提交归档** → 写清说明，保持提交粒度合理（每轮代码修改都必须提交 git）

### 审查测试规范
- **编译测试**：所有代码审查前必须执行构建命令进行编译测试，确保后端 + 前端均无编译错误。编译不通过则直接打回，不进入代码审查环节。
- **启动前清理**：如果上次构建进程未正常退出导致端口被占用，需先执行清理脚本清理残留进程后再重新启动。
- **审查流程**：开发完成后 → 编译测试 → 业务审查（功能正确性） → 代码审查（架构与规范）

## 多智能体团队角色（v2）

协调者通过 `multi-agent-dev-team` 技能编排以下 6 个专业角色，**严格按此顺序串行执行，禁止跳步或合并**：

| 简写 | 角色 | 职责 |
|------|------|------|
| **PM** | Project Manager | 需求拆解、制定执行计划 |
| **TD** | Tech Director | 评判可行性、技术选型、架构与规范设计 |
| **FE** | Frontend Developer | TypeScript / HTML / CSS 实现 |
| **BE** | Backend Developer | Rust 后端实现 |
| **BIZ-QA** | Business Reviewer | 审查功能是否符合业务要求 |
| **CODE-QA** | Code Reviewer | 审查代码质量、架构与规范遵守 |

### 串行质量闸门

```
PM → TD → (FE + BE 并行) → BIZ-QA → CODE-QA → 交付
          ↓                          ↓
     编译通过验收              编译通过验收
```

### 返工规则
- **BIZ-QA / CODE-QA 驳回**：不重新启动完整流程。协调者将审查报告原样转发至开发者，要求只修复所指问题，修复后重新进入 QA 审查。如果QA审查发现代码规范违背规范，则驳回修改，将报告原样转发至开发者。
- **连续 3 次驳回**：暂停流程，发出"自动修复失败，需要人工介入"警报。
