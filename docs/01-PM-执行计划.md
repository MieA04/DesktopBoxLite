### Sprint 1：核心窗口与图标网格

**目标**：实现全屏透明窗口、可拖拽调整大小、响应式网格布局

### 4.2 前端架构说明

- **无框架策略**：前端使用原生 TypeScript 与 HTML/CSS，不引入 React/Vue 等重型前端框架，保持极简轻量。
- **组件化思路**：虽然无框架，但仍采用类/模块化的方式组织代码，每个组件一个 `.ts` 文件。
- **通信方式**：前端通过 `@tauri-apps/api/core` 的 `invoke` 调用 Rust 后端命令。

### 4.3 后端架构说明

- 每个 Rust 模块文件对应一组相关功能，保持单一职责。
- `main.rs` 负责窗口创建和 Tauri 应用初始化。
- `lib.rs` 导出所有模块，注册 Tauri commands。
- 通过 `tauri::Builder::plugin()` 注册 Tauri 插件（如 global-hotkey）。
- 配置由 `config.rs` 模块集中管理，全局通过 Rust 的线程安全共享状态访问。

### 5.2 每个 Sprint 的审查流程

每个 Sprint 完成后，执行以下审查流程：

```
开发完成
    │
    ▼
┌─────────────────────────┐
│  步骤 A：编译测试        │ ◄── 后端 `cargo build` + 前端 `npm run build`
│  ▼ 通过？               │     或 `cargo tauri build --debug`
│     是 → 进入步骤 B      │
│     否 → 打回开发者修复   │
└─────────────────────────┘
    │
    ▼
┌─────────────────────────┐
│  步骤 B：业务审查        │ ◄── BIZ-QA 按验收标准逐条核查
│  ▼ 通过？               │
│     是 → 进入步骤 C      │
│     否 → 打回开发者修复   │
└─────────────────────────┘
    │
    ▼
┌─────────────────────────┐
│  步骤 C：代码审查        │ ◄── CODE-QA 检查架构、规范、一致性
│  ▼ 通过？               │
│     是 → 本轮 Sprint 完成│
│     否 → 打回开发者修复   │
└─────────────────────────┘
    │
    ▼
  合并/交付
```

### 5.3 编译测试要求

| 维度 | 要求 | 命令 |
|------|------|------|
| Rust 后端编译 | 无 warning（建议）、无 error | `cargo build` |
| TypeScript 编译 | 无 error，类型检查通过 | `npx tsc --noEmit` |
| 完整构建 | 前后端统一构建成功 | `cargo tauri build --debug` |
| 启动测试 | 应用正常启动，功能可用 | `cargo tauri dev` |

### 5.4 返工规则

依据 AGENTS.md：
- BIZ-QA 或 CODE-QA 驳回时，不重新启动完整流程。审查报告原样转发给对应开发者（FE 或 BE），只修复所指问题。
- 修复后重新进入 QA 审查环节。
- 连续 3 次驳回 → 暂停流程，发出"自动修复失败，需要人工介入"警报。

### 5.5 代码提交规范

- 每个 Sprint 完成后必须提交 git。
- 提交粒度：每个 Sprint 至少一次提交，建议按任务粒度拆分多次提交。
- 提交信息格式：`[Sprint N] 模块: 变更说明`
- 禁止在注释和日志中使用 emoji 表情（遵循 AGENTS.md 规范）。

## 六、非功能需求的实现策略

| ID | 指标 | 目标值 | 实现策略 |

## 七、风险与对策

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| Tauri 2 的 `transparent` 窗口在 Windows 上存在渲染问题 | 核心功能受阻 | 低 | 备选方案：设置背景色为半透明 + 全屏窗口模拟透明 |
