## 一、现有设计问题根源

### 核心缺陷：轮询无变更检测

```
现有逻辑:
  setInterval(500ms) ──→ refreshIcons()
                              │
                              └── invoke("get_icons")
                                      │
                                      └── scan_desktop_icons()
                                              │
                                              ├── 扫描目录
                                              ├── 提取所有图标  ← 不管变没变
                                              └── 返回全部数据
```

**根本错误**：每 500ms **无条件**全量提取所有图标。没有先问"文件变了吗？"再决定要不要提取。

### 连锁问题

| 问题 | 根因 |
|------|------|
| 同步提取阻塞主线程 | 提取逻辑与 Tauri command 在同一线程 |
| 无并发控制 | `setInterval` 不管上次是否完成，到点就触发 |
| 无取消机制 | 后端已经在提取时，前端又发新请求，任务堆积 |
| 无缓存层 | 每次从零开始 COM+GDI+PNG |
| 无法应对增删改 | 删了文件下次扫描才发现，没有实时响应删除 |

### 2.2 新数据流时序

```
t=0ms    前端 poll: check_icons_changed("abc") → false
          ↓ (无变更，等待)
t=500ms  前端 poll: check_icons_changed("abc") → true
          ↓
         前端 markRefreshing()
         前端 invoke("refresh_icons")
          ↓ (立即返回)
t=502ms  后端: 检查 extraction_running → false
         后端: 设置 cancel_token + extraction_running = true
         后端: 启动提取线程
         提取线程: for each icon:
           check cancel → 未取消
           check cache → miss
           COM GetImage(256x256) → GDI → PNG → base64
           save to cache
           ↓
t=3000ms 提取完成 → update cached_icons
          ↓
         emit("icons-ready")
          ↓
         后端: extraction_running = false
          ↓
t=3020ms 前端收到事件:
          invoke("get_icons") → 得到数据
          render() → markIdle()
          ↓
t=3500ms 前端 poll: check_icons_changed("abc") → false (无变化)
          ...

--- 如果在提取过程中又发生了变化 ---

t=2000ms 提取线程还在处理第 50 个图标
         前端 poll: check_icons_changed("abc") → true (又有变化)
          ↓
         前端: isRefreshing = true → set pendingRefresh = true
          ↓
t=2500ms 提取线程完成 → emit("icons-ready")
         前端收到 → render()
          ↓
         前端: markIdle() → 发现 pendingRefresh → 立即触发新一轮
          ↓
         invoke("refresh_icons")
         后端: cancel_token.send(()) → 取消旧的
         启动新线程
```

```rust
    let json = serde_json::to_string(&metas)
```

不引入 blake3 依赖？可以用 Rust 标准库的 `std::collections::hash_map::DefaultHasher`：

```rust
use std::hash::{Hash, Hasher};

fn fingerprint(metas: &[IconMeta]) -> u64 {
    let mut hasher = std::collections::hash_map::DefaultHasher::new();
    metas.hash(&mut hasher);
    hasher.finish()
}
```

### 3.2 后端：新增模块 `src-tauri/src/icon_state.rs`

```rust
use std::sync::{Arc, Mutex, atomic::AtomicBool};
use std::sync::mpsc;

pub struct IconState {
    pub fingerprint: Mutex<String>,
    pub cached_icons: Mutex<Vec<DesktopIcon>>,
    pub extraction_running: AtomicBool,
    pub cancel_tx: Mutex<Option<mpsc::Sender<()>>>,
    pub cache: IconCache,
}

impl IconState {
    pub fn new(config_path: &Path) -> Self { ... }
    
    pub fn request_refresh(&self, app: tauri::AppHandle) {
        // 1. Cancel current
        if let Some(tx) = self.cancel_tx.lock().unwrap().take() {
            let _ = tx.send(());
        }
        
        // 2. Start new
        let (tx, rx) = mpsc::channel();
        *self.cancel_tx.lock().unwrap() = Some(tx);
        self.extraction_running.store(true, Ordering::SeqCst);
        
        let state = /* Arc to self */;
        thread::spawn(move || {
            let result = run_extraction(&state, &rx);
            // Update cached_icons
            *state.cached_icons.lock().unwrap() = result;
            state.extraction_running.store(false, Ordering::SeqCst);
            let _ = app.emit("icons-ready", ());
        });
    }
}
```

```rust
use icon_state::IconState;

pub fn run() {
    // ...
    tauri::Builder::default()
        .manage(IconState::new(&config::config_path()))
        .invoke_handler(tauri::generate_handler![
            check_icons_changed,
            refresh_icons,

#[tauri::command]
fn check_icons_changed(state: tauri::State<IconState>) -> Result<bool, String> {
    let current_fp = icons::compute_fingerprint()?;
    let known = state.fingerprint.lock().unwrap();
    Ok(&current_fp != &*known)
}

#[tauri::command]
async fn refresh_icons(app: tauri::AppHandle, state: tauri::State<'_, IconState>) -> Result<(), String> {
    // 先更新 fingerprint
    let fp = icons::compute_fingerprint()?;
    *state.fingerprint.lock().unwrap() = fp;
    // 启动后台提取
    state.request_refresh(app);
    Ok(())
}

#[tauri::command]
fn get_icons(state: tauri::State<IconState>) -> Result<Vec<DesktopIcon>, String> {
    let icons = state.cached_icons.lock().unwrap();
    Ok(icons.clone())
}
```

### 3.5 前端：新增 `PollManager`

```typescript
class PollManager {
    private intervalMs: number;
    private isRefreshing = false;
    private pendingRefresh = false;
    private onIconsUpdate: (icons: IconInfo[]) => void;
    private timer: ReturnType<typeof setInterval> | null = null;

    start(): void {
        this.timer = setInterval(() => this.poll(), this.intervalMs);
    }

    stop(): void {
        if (this.timer) clearInterval(this.timer);
    }

    private async poll(): Promise<void> {
        try {
            // Step 1: 轻量轮询 — 变更检测
            const changed = await invoke<boolean>("check_icons_changed");
            if (!changed) return;

            // Step 2: 边界条件 — 防止重渲染冲突
            if (this.isRefreshing) {
                this.pendingRefresh = true;
                return;  // 下一轮 poll 会自动处理
            }

            // Step 3: 异步触发刷新
            this.isRefreshing = true;
            await invoke("refresh_icons");  // 后台线程处理，立即返回

            // Step 4: 等待后端完成事件
            await this.waitForReady();

            // Step 5: 获取缓存结果
            const icons = await invoke<IconInfo[]>("get_icons");
            this.onIconsUpdate(icons);
        } catch (err) {
            console.error("Poll refresh failed:", err);
        } finally {
            this.isRefreshing = false;
            if (this.pendingRefresh) {
                this.pendingRefresh = false;
                // 在提取过程中又有新的变更，下一轮 poll 会处理
            }
        }
    }

    private waitForReady(): Promise<void> {
        return new Promise((resolve) => {
            const unlisten = listen("icons-ready", () => {
                unlisten.then(fn => fn());
                resolve();
            });
        });
    }
}
```

## 六、边界条件与异常处理

| 场景 | 处理方式 |
|------|---------|
| 提取过程中文件被删 | 取消当前提取 → 下一轮检测到变更 → 重新提取 |
| 提取过程中文件被修改 | 同上 |
| 提取过程中应用退出 | cancel_token drop 时线程自动终止 |
| 缓存文件损坏 | `cache.get()` 解析失败 → 返回 None → 重新提取 |
| 缓存目录被手动删除 | `cache.set()` 重新创建目录 |
| 连续高频变更 | pendingRefresh 标记合并多次变更 → 只触发一次刷新 |
| 前端渲染耗时 > 500ms | renderGuard 阻止并发渲染 |

## 七、与原设计的对比

| 维度 | 原设计 | 新设计 |
|------|--------|--------|
| 轮询负载 | 全部图标提取（5s+） | 文件元数据扫描（~5ms） |
| 触发条件 | 无条件每 500ms | 仅 fingerprint 变更时 |
| 提取方式 | 同步阻塞主线程 | 后台线程 + cancel 支持 |
| 缓存 | 无 | 磁盘缓存（mtime 校验） |
| 并发控制 | 无（堆叠崩溃） | renderGuard + pendingRefresh |
| 增删改响应 | 500ms 后全量刷新 | 500ms 检测 + 仅处理变更 |
