## Issue 1：REQ-BOX-002 — 窗口可拖拽调整大小（需求修正）

### 现状分析

当前实现中：
- `tauri.conf.json` 设置 `"resizable": false`，OS 窗口本身不可缩放
- `ResizeHandle.ts` 仅修改 `.resizable-window` 的 CSS `width`/`height`
- **结果**：拖拽时 OS 窗口框架纹丝不动，仅内部内容 div 变化，完全违背需求语义

### 根因

原始需求文档 REQ-BOX-002 的"实现方式"描述不精确：
> "前端监听鼠标事件，动态修改窗口 width/height，并触发布局重新计算"

其中的"窗口 width/height"被前端团队理解为 CSS 样式中的宽高属性，而非 OS 窗口物理尺寸。这是 PM 需求表述歧义导致的实现偏差。

### 需求修正

| 字段 | 原内容 | 修正后 |
|------|--------|--------|
| **实现方式** | 前端监听鼠标事件，动态修改窗口 `width`/`height`，并触发布局重新计算 | 前端监听鼠标拖拽事件，**通过 Tauri `Window.setSize()` API 改变 OS 窗口的实际物理尺寸**。`.resizable-window` 内部容器始终填充 100% 窗口宽高，由 OS 窗口尺寸变化驱动 flexbox 布局自适应 |
| **验收标准 1** | 窗口四边及右下角支持拖拽缩放 | 窗口四边及右下角支持拖拽缩放，缩放时 **OS 窗口框架尺寸同步变化** |
| **新增验收标准 4** | — | **W（左）/ N（上）方向拖拽时，窗口位置同步调整**，保持右/下边缘不动（等效于自然拖拽体验） |
| **新增验收标准 5** | — | 拖拽缩放性能流畅，不出现明显卡顿或闪烁 |

### 技术约束

- 保持 `"resizable": false`（避免原生窗口装饰干扰），缩放完全通过程序化 `setSize()` 实现
- `@tauri-apps/api/window` 提供 `getCurrentWindow().setSize()` 和 `setPosition()` API

## Issue 2：REQ-BOX-010（新增）— 窗口可拖拽移动

### 现状分析

当前窗口使用 `decorations: false`（无标题栏），用户无法通过常规方式拖拽移动窗口。前期实现尝试在搜索栏添加 `data-tauri-drag-region` 属性，但搜索栏内的 `<input>` 输入框吞噬鼠标事件，导致拖拽不稳定、成功率低。

### 新需求

| 字段 | 内容 |
|------|------|
| **ID** | REQ-BOX-010 |
| **标题** | 窗口可拖拽移动 |
| **优先级** | P0 |
| **描述** | 用户可通过拖拽窗口特定区域（非内容操作区）来移动整个收纳盒窗口在屏幕上的位置 |
| **实现方式** | 在搜索栏顶部设置一条 **12px 高度的专用拖拽手柄条**（带有 `data-tauri-drag-region` 属性），该区域独立于搜索输入框，互不干扰 |
| **验收标准 1** | 窗口顶部有明显可拖拽区域（视觉提示：拖拽光标 + 背景色区分） |
| **验收标准 2** | 拖拽时窗口实时跟随鼠标移动，无延迟 |
| **验收标准 3** | 拖拽操作与搜索框输入功能完全隔离（点击拖拽条不会误触输入框） |
| **验收标准 4** | 窗口位置随启动保持底部居中（每次启动自动计算并设置位置） |
